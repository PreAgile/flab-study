# 12. Dynamic Programming (동적 계획법)

> "DP = 메모이제이션 쓰는 거" 라고 답하면 입문자.
> 마스터는 문제를 보자마자 **(1) 최적 부분 구조가 성립하는가**, **(2) 중복 부분 문제가 발생하는가**를 30초 안에 검증하고, **상태(state)를 무엇으로 잡을지** 정의한 뒤 점화식을 종이에 먼저 쓴다. Top-down vs Bottom-up은 디버깅 편의의 문제일 뿐 본질은 같다. LIS의 patience sort, Knapsack의 2D 테이블, Edit Distance의 INSERT/DELETE/REPLACE 3원자 연산까지 백지에서 그려낼 수 있어야 한다.
>
> 이 문서는 LeetCode/프로그래머스에서 자주 나오는 8개 카테고리(선형 DP, 그리드 DP, LIS류, LCS류, Knapsack류, 구간 DP, Bitmask DP, Kadane)를 본질·점화식·운영 매핑까지 한 번에 다룬다.

## 0. 인지 신호 — "이 문제, DP다"

문제를 30초 안에 DP로 분류하는 시그널.

| 신호 | 예시 문구 | 어떤 DP로? |
|---|---|---|
| **"최댓값/최솟값/개수를 구하라"** | "동전으로 만들 수 있는 최소 개수", "경로의 수" | 거의 항상 DP 후보 |
| **"선택지에 따른 분기"** | "이 집을 털지 말지", "이 동전을 쓸지 말지" | 0-1 결정 DP, House Robber/Knapsack |
| **단조 증가/감소 부분 수열** | "가장 긴 증가 부분 수열", "가장 긴 진동 수열" | LIS류, `dp[i] = 1 + max(dp[j])` |
| **두 문자열 비교** | "공통 부분 수열", "편집 거리", "regex 매칭" | LCS류, 2D DP |
| **그리드의 경로 수/최소합** | "좌상→우하 경로의 최소합/개수", "장애물 있는 경로" | Grid DP, `dp[i][j] = dp[i-1][j] + dp[i][j-1]` |
| **"X로 합을 만들 수 있는가/방법 수"** | "주어진 합을 만드는 동전 조합", "부분집합으로 절반 가르기" | Knapsack류 |
| **"연속된 부분 배열의 최대합"** | Maximum Subarray | Kadane (선형 DP) |
| **"구간을 합치거나 자르는 비용"** | "행렬 곱셈 순서", "팰린드롬 분할" | 구간 DP, `dp[l][r]` |
| **N이 작음 (N ≤ 20)** | "도시 N개 순회 최소 비용" | Bitmask DP, TSP |
| **재귀로 풀었더니 같은 인자가 계속 등장** | `solve(i-1) + solve(i-2)` 호출이 폭증 | 메모이제이션으로 변환 |

### 0.1 그리디·완전탐색과 어떻게 구분하나

| 패턴 | 핵심 | DP와 차이 |
|---|---|---|
| **Greedy** | 매 단계 지역 최적이 전역 최적 | DP는 모든 선택지를 탐색 후 결정, 그리디는 한 선택지만 본다 |
| **Brute Force / Backtracking** | 모든 가능성 탐색 | DP는 같은 부분 문제를 **한 번만** 푼다 (memo/table) |
| **Divide & Conquer** | 부분 문제로 쪼개되 **부분 문제가 겹치지 않음** (예: merge sort, quick sort) | DP는 부분 문제가 **겹친다** |

**핵심 시그널**: 재귀 호출 트리를 그려봤더니 같은 `(i, j)` 노드가 여러 번 나오면 → DP. 그렇지 않으면 → D&C.

### 0.2 시니어 운영 관점에서의 DP

DP는 학교 알고리즘이 아니다. production에서 다음과 같이 광범위하게 쓰인다.

| 영역 | DP 응용 |
|---|---|
| **DB Query Optimizer** | Join 순서 결정 — Selinger의 PostgreSQL/Oracle은 left-deep tree DP로 N개 테이블 조인 비용 최소화. `dp[S] = S 부분집합의 최소 join 비용`, Bitmask DP의 정석 |
| **CDN/Cache replacement** | Belady's OPT(미래를 아는 최적)는 DP로 lower bound 계산. 실제 LRU/LFU/ARC 평가 시 OPT 대비 hit rate 측정 |
| **Resource Allocation** | Kubernetes scheduler의 bin packing, Knapsack 변형 — N개 Pod를 M개 Node에 자원 제약 하에 배치 |
| **NLP/Bioinformatics** | DNA sequence alignment(Needleman-Wunsch), spell checker(Edit Distance) — 모두 LCS/Edit Distance 직접 변형 |
| **Network Routing** | Bellman-Ford는 DP (k번 이하 간선으로 도달하는 최단거리) |
| **Compiler** | DAG-based instruction scheduling, register allocation의 일부 |

→ "DP = 코딩 테스트용"이 아니라 **자원 최적화·비교·매칭이 필요한 모든 영역의 공통 도구**.

## 1. 백지 그리기 — DP를 종이에서 풀어내는 흐름

### 1.1 DP 문제 푸는 표준 5단계

```
[1] 상태(state) 정의 — dp[i] 또는 dp[i][j]가 정확히 무엇을 의미하는가?
      예: dp[i] = "i번째 원소까지 봤을 때 최대 합"
          dp[i][w] = "i번째 물건까지 고려, 무게 w 한도에서 최대 가치"
              │
              ▼
[2] 점화식(recurrence) — dp[i]를 dp[i-1], dp[i-2], ... 로 어떻게 표현하나?
      예: dp[i] = max(dp[i-1], dp[i-2] + nums[i])
              │
              ▼
[3] Base case — dp[0], dp[1] 등 가장 작은 부분 문제의 답
      예: dp[0] = nums[0], dp[1] = max(nums[0], nums[1])
              │
              ▼
[4] 전이 방향 — 어떤 순서로 채워야 의존성이 깨지지 않나?
      i 순방향? 역방향? i,j 모두 순방향? 대각선?
              │
              ▼
[5] 공간 최적화 — 필요한 이전 행/원소만 유지하면 O(n²)→O(n), O(n)→O(1)
```

### 1.2 Top-down vs Bottom-up

```
┌──────────────────────────────────────────────────────────────┐
│ Top-down (메모이제이션 = 재귀 + memo)                          │
│                                                              │
│   solve(5)                                                   │
│     ├─ solve(4)                                              │
│     │    ├─ solve(3)                                         │
│     │    │    ├─ solve(2) ─ memo[2] 저장                     │
│     │    │    └─ solve(1)                                    │
│     │    └─ solve(2) ─ memo[2] 재사용 (재계산 X)             │
│     └─ solve(3) ─ memo[3] 재사용                             │
│                                                              │
│  장점: 점화식 그대로 코드, 필요한 상태만 계산                │
│  단점: 재귀 stack overflow, 함수 호출 오버헤드               │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ Bottom-up (타뷸레이션 = 반복 + table)                         │
│                                                              │
│   dp[0] = ?                                                  │
│   dp[1] = ?                                                  │
│   for i in 2..n:                                             │
│       dp[i] = f(dp[i-1], dp[i-2], ...)                       │
│   return dp[n]                                               │
│                                                              │
│  장점: stack overflow X, cache-friendly, 공간 최적화 쉬움    │
│  단점: 모든 상태 계산 (불필요한 것도), 의존성 순서 설계 필요 │
└──────────────────────────────────────────────────────────────┘
```

**언제 어느 것?**
- **Top-down**: 상태 공간이 크지만 실제 도달하는 상태가 적을 때, 점화식이 자연스러운 재귀일 때 (트리, 그래프 DP)
- **Bottom-up**: 모든 상태를 어차피 계산할 때, 공간 최적화가 중요할 때 (Knapsack, LIS)

### 1.3 1D DP 채우기 (Climbing Stairs)

```
n = 5, dp[i] = i번째 계단에 도달하는 방법의 수
점화식: dp[i] = dp[i-1] + dp[i-2]
base:   dp[0] = 1, dp[1] = 1

i:    0  1  2  3  4  5
dp:   1  1  ?  ?  ?  ?
         ↓
i=2: dp[2] = dp[1] + dp[0] = 1+1 = 2
i=3: dp[3] = dp[2] + dp[1] = 2+1 = 3
i=4: dp[4] = dp[3] + dp[2] = 3+2 = 5
i=5: dp[5] = dp[4] + dp[3] = 5+3 = 8

dp:   1  1  2  3  5  8

→ 공간 최적화: dp[i]는 dp[i-1], dp[i-2]만 보면 됨 → O(1) 공간으로 압축
   prev2, prev1 두 변수만 유지
```

### 1.4 2D DP 채우기 (LCS / Edit Distance)

```
LCS: text1 = "abcde", text2 = "ace"
dp[i][j] = text1[0..i-1]과 text2[0..j-1]의 LCS 길이
점화식:
   text1[i-1] == text2[j-1] ─▶ dp[i][j] = dp[i-1][j-1] + 1
   text1[i-1] != text2[j-1] ─▶ dp[i][j] = max(dp[i-1][j], dp[i][j-1])

        ""  a   c   e
    "" [ 0  0   0   0 ]
    a  [ 0  1   1   1 ]
    b  [ 0  1   1   1 ]
    c  [ 0  1   2   2 ]
    d  [ 0  1   2   2 ]
    e  [ 0  1   2   3 ]  ← 답: 3 ("ace")

전이 방향:
    ┌─────┬─────┐
    │ ↖   │ ↑   │     ↖ : 대각선 (문자 같을 때)
    ├─────┼─────┤     ↑ : 위 (text1 한 글자 무시)
    │ ←   │  ●  │     ← : 왼쪽 (text2 한 글자 무시)
    └─────┴─────┘
→ 의존성: 왼쪽·위·대각선만 본다 → 행 순서대로, 각 행은 왼→오 채우면 안전
→ 공간 최적화: 직전 행만 필요 → O(min(m, n)) 공간
```

### 1.5 0-1 Knapsack 2D 테이블 채우기

```
물건: (w=2,v=3), (w=3,v=4), (w=4,v=5), (w=5,v=6)
용량 W = 5
dp[i][w] = i번째 물건까지 고려, 무게 ≤ w 일 때 최대 가치
점화식:
   현재 물건 무게 w_i > w ─▶ dp[i][w] = dp[i-1][w]   (못 담음)
   그 외:
     dp[i][w] = max(dp[i-1][w],                     (안 담음)
                    dp[i-1][w - w_i] + v_i)         (담음)

        w=0  1   2   3   4   5
   i=0 [ 0   0   0   0   0   0 ]
   i=1 [ 0   0   3   3   3   3 ]   ← (2,3) 추가
   i=2 [ 0   0   3   4   4   7 ]   ← (3,4) 추가, dp[2][5] = max(3, dp[1][2]+4=7)
   i=3 [ 0   0   3   4   5   7 ]   ← (4,5) 추가
   i=4 [ 0   0   3   4   5   7 ]   ← (5,6) 추가, w=5에서 6 vs 7 → 7 유지

답: dp[4][5] = 7

공간 최적화: dp[i][w]는 dp[i-1][...]만 본다 → 1D로 압축
   주의: w를 **역방향**(W → w_i)으로 순회해야 같은 행에서 dp[i-1] 값을 덮어쓰지 않음
   (정방향 순회는 무한 개수 Knapsack, 같은 물건 여러 번 담는 효과)
```

### 1.6 LIS — Patience Sort 시각화 (O(n log n))

```
arr = [10, 9, 2, 5, 3, 7, 101, 18]
LIS = [2, 3, 7, 101] 또는 [2, 3, 7, 18] (길이 4)

Patience sort 방식: 카드를 더미(pile)에 놓되, 자기보다 작은 더미 top 위에 놓는다.
                   자기보다 작은 더미가 없으면 새 더미 생성.
                   더미 개수 = LIS 길이.

원소: 10        9        2        5         3         7          101         18
       │        │        │        │         │         │           │           │
       ▼        ▼        ▼        ▼         ▼         ▼           ▼           ▼
더미: [10]    [9]      [2]      [2]       [2]       [2]         [2]         [2]
              [10X→9]  [10X→9]  [5]       [3]       [3]         [3]         [3]
                       [9 X→2]  [9 X→5]   [9 X→5]   [7]         [7]         [7]
                                          [10X→5]→? [101]       [101]       [18]
                                                                              [101→18]

각 단계에서 새 원소 x는 "x보다 큰 첫 번째 더미 top"을 찾아 그 위에 놓음 (binary search).
→ tails[] 배열 = 각 더미의 top 값을 유지
   tails = [2, 3, 7, 18] 일 때 길이 4 = LIS 길이

핵심: tails는 정렬된 상태가 유지됨 → lowerBound binary search로 O(log n)
주의: tails 자체는 실제 LIS가 아님 (길이만 정확). 실제 부분수열 복원하려면 별도 추적 필요.
```

## 2. 직관과 정의

### 2.1 한 줄 비유

> **"같은 계산을 또 하지 마라" 원칙**. 부분 문제의 답을 노트(memo/table)에 적어두고, 다시 필요하면 노트를 본다. 그 노트가 다음 부분 문제의 답을 만드는 데도 쓰이는 것이 DP.

### 2.2 정확한 정의 (Bellman, 1957)

DP가 적용 가능하려면 문제가 두 성질을 만족해야 한다.

| 성질 | 의미 | 위배되면 |
|---|---|---|
| **Optimal Substructure (최적 부분 구조)** | 큰 문제의 최적해를 작은 부분 문제의 최적해로부터 구성 가능 | DP 자체가 성립 안 함 → 다른 방법 (greedy, brute force) |
| **Overlapping Subproblems (중복 부분 문제)** | 같은 부분 문제가 여러 번 등장 | Divide & Conquer로 충분 (memo 불필요) |

**예시 — 둘 다 성립**: Fibonacci, LCS, Knapsack
**최적 부분 구조 있으나 중복 X**: Merge sort, Quick sort (D&C로 분류)
**최적 부분 구조 없음**: Longest Path in general graph (NP-hard, DAG라면 가능)

### 2.3 상태(state) 정의가 본질

DP 문제의 90%는 상태 정의에서 결정된다. 잘못 잡으면 점화식이 안 나오거나, 차원이 폭발한다.

| 문제 | 상태 정의 |
|---|---|
| Climbing Stairs | `dp[i]` = i번째 계단까지의 방법 수 |
| House Robber | `dp[i]` = i번째 집까지 봤을 때 최대 도난액 |
| Coin Change | `dp[a]` = 금액 a를 만드는 최소 동전 개수 |
| LIS | `dp[i]` = i번째 원소를 **마지막으로 포함하는** LIS 길이 |
| LCS | `dp[i][j]` = text1[0..i-1]과 text2[0..j-1]의 LCS 길이 |
| 0-1 Knapsack | `dp[i][w]` = i번째 물건까지 고려, 무게 ≤ w 일 때 최대 가치 |
| Edit Distance | `dp[i][j]` = word1[0..i-1] → word2[0..j-1] 변환 최소 연산 수 |
| Bitmask DP (TSP) | `dp[mask][i]` = 방문한 도시 집합 mask, 현재 위치 i일 때 최소 비용 |

**상태 정의 체크리스트**:
1. `dp[?]`의 ? 부분이 문제의 "현재 상황"을 완전히 표현하는가? (Markov 성질 — 과거 경로 무관, 현재 상태만으로 미래 결정 가능)
2. 답이 `dp[n]` 또는 `dp[n][m]` 같은 명확한 위치에 있는가?
3. 상태 수 × 전이 비용이 시간 제한 안에 들어오는가?

### 2.4 상태 압축 — "이전 행만 필요"

대부분의 2D DP는 `dp[i][...]`가 `dp[i-1][...]`만 참조한다. 즉, 행 전체를 들고 있을 필요 없이 **2개 행 또는 1개 행**만 유지하면 된다.

```
원본:                   압축:
dp[N][M]                prev[M], curr[M]    또는 1개 행만 in-place
공간 O(N×M)             공간 O(M)
```

Knapsack 1D 압축은 추가 트릭이 필요 — **무게 w를 역방향으로** 순회해야 같은 물건 중복 사용을 방지(0-1). 무한 개수(Unbounded)면 정방향.

## 3. Java 템플릿

### 3.1 Fibonacci — DP의 Hello World

```java
// Top-down (memoization)
class FibTopDown {
    private long[] memo;

    public long fib(int n) {
        memo = new long[n + 1];
        Arrays.fill(memo, -1);
        return solve(n);
    }

    private long solve(int n) {
        if (n <= 1) return n;
        if (memo[n] != -1) return memo[n];
        return memo[n] = solve(n - 1) + solve(n - 2);
    }
}

// Bottom-up (tabulation)
class FibBottomUp {
    public long fib(int n) {
        if (n <= 1) return n;
        long[] dp = new long[n + 1];
        dp[0] = 0; dp[1] = 1;
        for (int i = 2; i <= n; i++) {
            dp[i] = dp[i - 1] + dp[i - 2];
        }
        return dp[n];
    }
}

// Bottom-up + O(1) space
class FibOptimized {
    public long fib(int n) {
        if (n <= 1) return n;
        long prev2 = 0, prev1 = 1;
        for (int i = 2; i <= n; i++) {
            long curr = prev1 + prev2;
            prev2 = prev1;
            prev1 = curr;
        }
        return prev1;
    }
}
```

### 3.2 Coin Change — 1D DP, 최솟값

```java
class CoinChange {
    public int coinChange(int[] coins, int amount) {
        int[] dp = new int[amount + 1];
        // dp[a] = a 금액을 만드는 최소 동전 개수
        // 불가능 표시를 위해 큰 값으로 초기화 (amount+1은 가능한 최대 동전 수 + 1)
        Arrays.fill(dp, amount + 1);
        dp[0] = 0;
        for (int a = 1; a <= amount; a++) {
            for (int coin : coins) {
                if (coin <= a) {
                    dp[a] = Math.min(dp[a], dp[a - coin] + 1);
                }
            }
        }
        return dp[amount] > amount ? -1 : dp[amount];
    }
}
```

핵심 트릭: 불가능을 -1로 표시하면 점화식에서 비교가 꼬인다. **"도달 불가" 값을 큰 값(amount+1)으로 두고** 마지막에만 -1로 변환.

### 3.3 0-1 Knapsack — 2D DP + 1D 공간 압축

```java
class Knapsack01 {
    // 2D 버전
    public int knapsack2D(int[] weights, int[] values, int W) {
        int n = weights.length;
        int[][] dp = new int[n + 1][W + 1];
        for (int i = 1; i <= n; i++) {
            for (int w = 0; w <= W; w++) {
                if (weights[i - 1] > w) {
                    dp[i][w] = dp[i - 1][w];
                } else {
                    dp[i][w] = Math.max(
                        dp[i - 1][w],
                        dp[i - 1][w - weights[i - 1]] + values[i - 1]
                    );
                }
            }
        }
        return dp[n][W];
    }

    // 1D 공간 압축 — 0-1 (역방향 순회 필수)
    public int knapsack1D(int[] weights, int[] values, int W) {
        int n = weights.length;
        int[] dp = new int[W + 1];
        for (int i = 0; i < n; i++) {
            // w를 W → weights[i] 역방향: 같은 물건 중복 방지
            for (int w = W; w >= weights[i]; w--) {
                dp[w] = Math.max(dp[w], dp[w - weights[i]] + values[i]);
            }
        }
        return dp[W];
    }

    // 무한 개수 Knapsack — 정방향 순회 (같은 물건 여러 번 OK)
    public int knapsackUnbounded(int[] weights, int[] values, int W) {
        int n = weights.length;
        int[] dp = new int[W + 1];
        for (int i = 0; i < n; i++) {
            for (int w = weights[i]; w <= W; w++) {  // 정방향
                dp[w] = Math.max(dp[w], dp[w - weights[i]] + values[i]);
            }
        }
        return dp[W];
    }
}
```

**왜 역방향?** 1D에서 `dp[w - weights[i]]`는 `dp[i-1][w - weights[i]]`(이전 행)이어야 한다. 정방향 순회는 `dp[w - weights[i]]`가 이미 `dp[i][w - weights[i]]`(현재 행)로 갱신된 후라서 같은 물건 중복 사용. 역방향은 작은 w가 아직 안 건드려진 상태라 안전.

### 3.4 LIS — O(n²) 그리고 O(n log n)

```java
class LongestIncreasingSubsequence {
    // O(n^2) — DP
    public int lengthOfLIS_DP(int[] nums) {
        int n = nums.length;
        int[] dp = new int[n];
        // dp[i] = i번째 원소를 마지막으로 포함하는 LIS 길이
        Arrays.fill(dp, 1);
        int best = 1;
        for (int i = 1; i < n; i++) {
            for (int j = 0; j < i; j++) {
                if (nums[j] < nums[i]) {
                    dp[i] = Math.max(dp[i], dp[j] + 1);
                }
            }
            best = Math.max(best, dp[i]);
        }
        return best;
    }

    // O(n log n) — Patience Sort + Binary Search
    public int lengthOfLIS_NlogN(int[] nums) {
        int[] tails = new int[nums.length];
        int size = 0;
        for (int x : nums) {
            int lo = 0, hi = size;
            // tails에서 x 이상의 첫 위치 (lower bound)
            while (lo < hi) {
                int mid = (lo + hi) >>> 1;
                if (tails[mid] < x) lo = mid + 1;
                else hi = mid;
            }
            tails[lo] = x;
            if (lo == size) size++;
        }
        return size;
    }
}
```

**주의**: O(n log n) 알고리즘의 `tails` 배열은 LIS 길이만 정확하고, **실제 LIS 부분수열이 아니다**. 부분수열 복원이 필요하면 별도 인덱스 추적이 필요.

### 3.5 LCS — 2D DP

```java
class LongestCommonSubsequence {
    public int longestCommonSubsequence(String text1, String text2) {
        int m = text1.length(), n = text2.length();
        int[][] dp = new int[m + 1][n + 1];
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (text1.charAt(i - 1) == text2.charAt(j - 1)) {
                    dp[i][j] = dp[i - 1][j - 1] + 1;
                } else {
                    dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
                }
            }
        }
        return dp[m][n];
    }

    // 공간 압축: O(min(m,n))
    public int longestCommonSubsequence_O1Row(String text1, String text2) {
        if (text1.length() < text2.length()) {
            String tmp = text1; text1 = text2; text2 = tmp;
        }
        int m = text1.length(), n = text2.length();
        int[] prev = new int[n + 1];
        int[] curr = new int[n + 1];
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (text1.charAt(i - 1) == text2.charAt(j - 1)) {
                    curr[j] = prev[j - 1] + 1;
                } else {
                    curr[j] = Math.max(prev[j], curr[j - 1]);
                }
            }
            int[] tmp = prev; prev = curr; curr = tmp;
            Arrays.fill(curr, 0);
        }
        return prev[n];
    }
}
```

### 3.6 Edit Distance — 3원자 연산

```java
class EditDistance {
    public int minDistance(String word1, String word2) {
        int m = word1.length(), n = word2.length();
        int[][] dp = new int[m + 1][n + 1];
        // dp[i][j] = word1[0..i-1] → word2[0..j-1] 최소 연산 수
        for (int i = 0; i <= m; i++) dp[i][0] = i;  // 전부 삭제
        for (int j = 0; j <= n; j++) dp[0][j] = j;  // 전부 삽입
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (word1.charAt(i - 1) == word2.charAt(j - 1)) {
                    dp[i][j] = dp[i - 1][j - 1];
                } else {
                    dp[i][j] = 1 + Math.min(
                        dp[i - 1][j - 1],  // REPLACE
                        Math.min(
                            dp[i - 1][j],  // DELETE word1[i-1]
                            dp[i][j - 1]   // INSERT word2[j-1]
                        )
                    );
                }
            }
        }
        return dp[m][n];
    }
}
```

3원자 연산:
- **INSERT** (`dp[i][j-1] + 1`) — word2의 j번째 글자 삽입
- **DELETE** (`dp[i-1][j] + 1`) — word1의 i번째 글자 삭제
- **REPLACE** (`dp[i-1][j-1] + 1`) — 같은 글자면 0, 다르면 1

이 3가지가 모든 문자열 편집 문제의 원자 단위. Levenshtein distance라고도 부르며 NLP spell checker, DNA alignment(Needleman-Wunsch는 변형)의 기반.

### 3.7 Grid DP — Minimum Path Sum

```java
class MinimumPathSum {
    public int minPathSum(int[][] grid) {
        int m = grid.length, n = grid[0].length;
        int[][] dp = new int[m][n];
        dp[0][0] = grid[0][0];
        for (int i = 1; i < m; i++) dp[i][0] = dp[i - 1][0] + grid[i][0];
        for (int j = 1; j < n; j++) dp[0][j] = dp[0][j - 1] + grid[0][j];
        for (int i = 1; i < m; i++) {
            for (int j = 1; j < n; j++) {
                dp[i][j] = grid[i][j] + Math.min(dp[i - 1][j], dp[i][j - 1]);
            }
        }
        return dp[m - 1][n - 1];
    }

    // In-place — grid를 그대로 dp로 사용
    public int minPathSumInPlace(int[][] grid) {
        int m = grid.length, n = grid[0].length;
        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++) {
                if (i == 0 && j == 0) continue;
                else if (i == 0) grid[i][j] += grid[i][j - 1];
                else if (j == 0) grid[i][j] += grid[i - 1][j];
                else grid[i][j] += Math.min(grid[i - 1][j], grid[i][j - 1]);
            }
        }
        return grid[m - 1][n - 1];
    }
}
```

### 3.8 Bitmask DP — TSP 스타일 (간단 버전)

```java
class TSP {
    // n개 도시, 도시 0에서 시작해 모든 도시를 방문하고 0으로 돌아오는 최소 비용
    public int tsp(int[][] dist) {
        int n = dist.length;
        int[][] dp = new int[1 << n][n];
        for (int[] row : dp) Arrays.fill(row, Integer.MAX_VALUE / 2);
        dp[1][0] = 0;  // 도시 0만 방문, 위치 = 0

        for (int mask = 1; mask < (1 << n); mask++) {
            for (int u = 0; u < n; u++) {
                if ((mask & (1 << u)) == 0) continue;
                if (dp[mask][u] >= Integer.MAX_VALUE / 2) continue;
                for (int v = 0; v < n; v++) {
                    if ((mask & (1 << v)) != 0) continue;  // 이미 방문
                    int newMask = mask | (1 << v);
                    dp[newMask][v] = Math.min(dp[newMask][v], dp[mask][u] + dist[u][v]);
                }
            }
        }

        int full = (1 << n) - 1;
        int ans = Integer.MAX_VALUE;
        for (int u = 1; u < n; u++) {
            ans = Math.min(ans, dp[full][u] + dist[u][0]);
        }
        return ans;
    }
}
```

상태: `dp[mask][u]` = 방문한 도시 집합 mask, 현재 위치 u일 때 최소 비용.
시간: O(2^n × n²), 공간: O(2^n × n). n ≤ 20에서만 실용적.

## 5. 시간/공간 복잡도

### 5.1 일반 공식

> **시간 = 상태 수 × 전이 비용**
> **공간 = 상태 수 (압축 가능 여부 별도 분석)**

| 문제 | 상태 수 | 전이 비용 | 시간 | 공간 |
|---|---|---|---|---|
| Fibonacci, Climbing Stairs | O(n) | O(1) | O(n) | O(n) → O(1) |
| House Robber | O(n) | O(1) | O(n) | O(n) → O(1) |
| Coin Change | O(amount) | O(coins) | O(amount × coins) | O(amount) |
| LIS (DP) | O(n) | O(n) | O(n²) | O(n) |
| LIS (Patience+BS) | O(n) | O(log n) | O(n log n) | O(n) |
| LCS, Edit Distance | O(m × n) | O(1) | O(m × n) | O(m × n) → O(min(m,n)) |
| 0-1 Knapsack | O(n × W) | O(1) | O(n × W) | O(n × W) → O(W) |
| Grid DP | O(m × n) | O(1) | O(m × n) | O(m × n) → O(n) |
| Bitmask DP (TSP) | O(2^n × n) | O(n) | O(2^n × n²) | O(2^n × n) |
| Kadane (MaxSubarray) | O(n) | O(1) | O(n) | O(1) |

### 5.2 Knapsack의 "pseudo-polynomial" 함정

0-1 Knapsack은 O(n × W)다. 이는 **W가 입력의 값이지 비트 수가 아니다.** 입력 크기 = O(n log W) (W를 표현하는 비트). 따라서 W가 2^32에 가까우면 입력 크기는 작아도 시간은 폭발 — pseudo-polynomial. NP-hard인 이유.

실전 의미: 코딩 테스트에서 `W ≤ 10^4`이면 OK. `W ≤ 10^9`면 DP로 안 풀린다 (별도 그리디·이진 탐색 접근 필요).

### 5.3 공간 최적화 표

```
원본 → 압축

dp[N]      → 변수 prev, curr 2개         (O(N) → O(1))
dp[N][M]   → dp[2][M] 또는 dp[M]         (O(NM) → O(M))
dp[N][N]   → 대각선 순회로 dp[N]         (구간 DP, 어려움)
dp[2^N][N] → 압축 어려움 (Bitmask)        (그대로 사용)
```

### 6.1 LeetCode 70 — Climbing Stairs

**요약**: n 계단을 한 번에 1 또는 2칸씩 올라갈 때, n까지 가는 방법의 수.
**점화식**: `dp[i] = dp[i-1] + dp[i-2]` (Fibonacci와 동치)
**Base**: `dp[0] = 1, dp[1] = 1` (또는 `dp[1] = 1, dp[2] = 2`)

```java
class Solution {
    public int climbStairs(int n) {
        if (n <= 2) return n;
        int prev2 = 1, prev1 = 2;
        for (int i = 3; i <= n; i++) {
            int curr = prev1 + prev2;
            prev2 = prev1;
            prev1 = curr;
        }
        return prev1;
    }
}
```

```kotlin

    var prev1 = 2
    for (i in 3..n) {
        val curr = prev1 + prev2
```

**복잡도**: O(n) 시간, O(1) 공간.
**함정**: n=1, n=2 base case. 한 번에 1, 2, 3칸 변형이면 `dp[i] = dp[i-1] + dp[i-2] + dp[i-3]`.

### 6.2 LeetCode 198 — House Robber

**요약**: 인접한 집을 털 수 없을 때 털 수 있는 최대 금액.
**상태**: `dp[i]` = i번째 집까지 봤을 때 최대 도난액 (i번째를 털었든 안 털었든)
**점화식**: `dp[i] = max(dp[i-1], dp[i-2] + nums[i])`
  - `dp[i-1]`: i번째를 안 털고 i-1까지의 최대
  - `dp[i-2] + nums[i]`: i번째를 털고 i-2까지의 최대 (i-1은 못 털음)

```java
class Solution {
    public int rob(int[] nums) {
        int n = nums.length;
        if (n == 1) return nums[0];
        int prev2 = nums[0];
        int prev1 = Math.max(nums[0], nums[1]);
        for (int i = 2; i < n; i++) {
            int curr = Math.max(prev1, prev2 + nums[i]);
            prev2 = prev1;
            prev1 = curr;
        }
        return prev1;
    }
}
```


- 변형 **House Robber III** (LeetCode 337, 트리) — 트리 DP, 각 노드에서 (털었을 때, 안 털었을 때) pair 반환.

### 6.3 LeetCode 322 — Coin Change

**요약**: 동전 종류 `coins`와 금액 `amount`. 금액을 만드는 최소 동전 수. 불가능하면 -1.
**상태**: `dp[a]` = a 금액을 만드는 최소 동전 수
**점화식**: `dp[a] = min(dp[a - coin] + 1) for coin in coins`
**Base**: `dp[0] = 0`, 나머지 `dp[a] = INF` (`amount + 1`)

```java
class Solution {
    public int coinChange(int[] coins, int amount) {
        int[] dp = new int[amount + 1];
        Arrays.fill(dp, amount + 1);
        dp[0] = 0;
        for (int a = 1; a <= amount; a++) {
            for (int coin : coins) {
                if (coin <= a) {
                    dp[a] = Math.min(dp[a], dp[a - coin] + 1);
                }
            }
        }
        return dp[amount] > amount ? -1 : dp[amount];
    }
}
```


**복잡도**: O(amount × coins.length) 시간, O(amount) 공간.
**함정**:
- 그리디로 풀면 안 된다. `coins = [1, 3, 4], amount = 6` → 그리디(큰 동전부터)는 4+1+1=3개, DP는 3+3=2개.
- 변형 **Coin Change II** (LeetCode 518, 조합 수) — `dp[a] += dp[a-coin]`, 단 **coin loop이 outer**여야 조합(순서 무관) 카운팅. 안쪽에 두면 순열 카운팅.

### 6.4 LeetCode 300 — Longest Increasing Subsequence (LIS)

**요약**: 가장 긴 증가 부분 수열의 길이.

**버전 1 — O(n²) DP**:
**상태**: `dp[i]` = i번째 원소를 마지막으로 포함하는 LIS 길이
**점화식**: `dp[i] = max(dp[j] + 1) for j < i if nums[j] < nums[i]`

```java
public int lengthOfLIS(int[] nums) {
    int n = nums.length;
    int[] dp = new int[n];
    Arrays.fill(dp, 1);
    int best = 1;
    for (int i = 1; i < n; i++) {
        for (int j = 0; j < i; j++) {
            if (nums[j] < nums[i]) dp[i] = Math.max(dp[i], dp[j] + 1);
        }
        best = Math.max(best, dp[i]);
    }
    return best;
}
```

**버전 2 — O(n log n) Patience Sort**:

```java
public int lengthOfLIS(int[] nums) {
    int[] tails = new int[nums.length];
    int size = 0;
    for (int x : nums) {
        int lo = 0, hi = size;
        while (lo < hi) {
            int mid = (lo + hi) >>> 1;
            if (tails[mid] < x) lo = mid + 1;
            else hi = mid;
        }
        tails[lo] = x;
        if (lo == size) size++;
    }
    return size;
}
```


**복잡도**: O(n²) 또는 O(n log n) 시간, O(n) 공간.
**함정**:
- **엄격 증가 vs 비감소** — `nums[j] < nums[i]` (엄격) vs `nums[j] <= nums[i]` (비감소). LIS의 O(n log n)에서는 lowerBound(첫 ≥) vs upperBound(첫 >) 차이.
- `tails` 배열은 LIS 길이만 정확, 실제 부분수열 아님.
- 변형 **Russian Doll Envelopes** (LeetCode 354) — 2차원 LIS, 한 축으로 정렬 후 다른 축으로 LIS. 정렬 시 동률은 두 번째 축 **내림차순** (같은 너비끼리는 LIS에 1개만).

### 6.5 LeetCode 1143 — Longest Common Subsequence (LCS)

**요약**: 두 문자열의 가장 긴 공통 부분 수열의 길이.
**상태**: `dp[i][j]` = text1[0..i-1]과 text2[0..j-1]의 LCS 길이
**점화식**:
- 마지막 글자 같음: `dp[i][j] = dp[i-1][j-1] + 1`
- 다름: `dp[i][j] = max(dp[i-1][j], dp[i][j-1])`

```java
public int longestCommonSubsequence(String text1, String text2) {
    int m = text1.length(), n = text2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            dp[i][j] = (text1.charAt(i - 1) == text2.charAt(j - 1))
                ? dp[i - 1][j - 1] + 1
                : Math.max(dp[i - 1][j], dp[i][j - 1]);
        }
    }
    return dp[m][n];
}
```


**함정**:
- **공통 부분 수열(Subsequence) vs 공통 부분 문자열(Substring)** 구분. Substring은 연속, Subsequence는 비연속 OK.
- 부분 수열 복원: dp 테이블을 역추적. `dp[i][j] == dp[i-1][j-1] + 1` 이면서 글자 같으면 그 글자가 LCS에 포함.
- 변형 **Shortest Common Supersequence** (LeetCode 1092) — `m + n - LCS` 길이.

### 6.6 LeetCode 53 — Maximum Subarray (Kadane)

**요약**: 연속된 부분 배열 중 합이 최대인 것.
**상태**: `current` = 현재 위치를 마지막으로 끝나는 부분 배열의 최대 합
**점화식**: `current = max(nums[i], current + nums[i])`
**핵심**: 현재까지 합이 음수면 버리고 새로 시작.

```java
public int maxSubArray(int[] nums) {
    int current = nums[0];
    int best = nums[0];
    for (int i = 1; i < nums.length; i++) {
        current = Math.max(nums[i], current + nums[i]);
        best = Math.max(best, current);
    }
    return best;
}
```

### 6.7 LeetCode 72 — Edit Distance

**요약**: word1을 word2로 바꾸는 최소 연산 수 (insert/delete/replace).
**상태**: `dp[i][j]` = word1[0..i-1] → word2[0..j-1] 최소 연산 수
**점화식**:
- `word1[i-1] == word2[j-1]`: `dp[i][j] = dp[i-1][j-1]`
- 다름: `dp[i][j] = 1 + min(dp[i-1][j-1], dp[i-1][j], dp[i][j-1])`
  - `dp[i-1][j-1] + 1`: REPLACE
  - `dp[i-1][j] + 1`: DELETE word1[i-1]
  - `dp[i][j-1] + 1`: INSERT word2[j-1]

```java
public int minDistance(String word1, String word2) {
    int m = word1.length(), n = word2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 0; i <= m; i++) dp[i][0] = i;
    for (int j = 0; j <= n; j++) dp[0][j] = j;
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (word1.charAt(i - 1) == word2.charAt(j - 1)) {
                dp[i][j] = dp[i - 1][j - 1];
            } else {
                dp[i][j] = 1 + Math.min(
                    dp[i - 1][j - 1],
                    Math.min(dp[i - 1][j], dp[i][j - 1])
                );
            }
        }
    }
    return dp[m][n];
}
```


**복잡도**: O(m × n) 시간 및 공간.
**함정**:

- 변형 가중치 다른 비용 (insert=1, delete=2, replace=3) — 점화식 weight 곱하면 됨.
- 운영 응용 — Spell checker, Git diff (Myers diff는 LCS 변형), DNA alignment.

### 6.8 LeetCode 416 — Partition Equal Subset Sum (0-1 Knapsack)

**요약**: 배열을 부분집합 두 개로 나눠 합이 같게 할 수 있는가?
**핵심 변환**: 총합이 짝수여야 하고, target = sum/2를 부분집합으로 만들 수 있는가? → **subset sum 문제**
**상태**: `dp[w]` = 무게 합 w를 만들 수 있는가? (boolean)
**점화식**: `dp[w] = dp[w] || dp[w - nums[i]]` (역방향 순회)

```java
public boolean canPartition(int[] nums) {
    int total = 0;
    for (int x : nums) total += x;
    if (total % 2 != 0) return false;
    int target = total / 2;
    boolean[] dp = new boolean[target + 1];
    dp[0] = true;
    for (int x : nums) {
        for (int w = target; w >= x; w--) {
            dp[w] = dp[w] || dp[w - x];
        }
        if (dp[target]) return true;  // 조기 종료
    }
    return dp[target];
}
```

```kotlin
fun canPartition(nums: IntArray): Boolean {
    val total = nums.sum()
    if (total % 2 != 0) return false
    val target = total / 2
    val dp = BooleanArray(target + 1)
    dp[0] = true
    for (x in nums) {
        for (w in target downTo x) {
            dp[w] = dp[w] || dp[w - x]
        }
        if (dp[target]) return true
    }
    return dp[target]
}
```

**복잡도**: O(n × target) 시간, O(target) 공간.
**함정**:
- 0-1이므로 **역방향 순회 필수** (정방향이면 같은 원소를 여러 번 쓰는 효과).
- 합이 홀수면 즉시 false.
- 변형 **Target Sum** (LeetCode 494) — `+/-` 기호로 target 만들기. `(P - N) = target, P + N = sum` → P = (sum + target)/2 → subset sum.

### 6.9 LeetCode 64 — Minimum Path Sum (Grid DP)

**요약**: m×n 그리드 좌상→우하 최소 경로 합 (오른쪽/아래만 이동).
**상태**: `dp[i][j]` = (0,0)→(i,j) 최소 합
**점화식**: `dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])`
**Base**: `dp[0][0] = grid[0][0]`, 첫 행/열은 누적합.

```java
public int minPathSum(int[][] grid) {
    int m = grid.length, n = grid[0].length;
    for (int i = 0; i < m; i++) {
        for (int j = 0; j < n; j++) {
            if (i == 0 && j == 0) continue;
            else if (i == 0) grid[i][j] += grid[i][j - 1];
            else if (j == 0) grid[i][j] += grid[i - 1][j];
            else grid[i][j] += Math.min(grid[i - 1][j], grid[i][j - 1]);
        }
    }
    return grid[m - 1][n - 1];
}
```

**복잡도**: O(m × n) 시간. In-place로 O(1) 추가 공간.
**함정**:
- 첫 행/열의 base case 빠뜨리면 IndexOutOfBounds.
- 변형 **Unique Paths** (LeetCode 62) — 경로 수만 세기: `dp[i][j] = dp[i-1][j] + dp[i][j-1]`.
- 변형 **Unique Paths II** (LeetCode 63) — 장애물 있는 경우, 장애물 칸은 `dp[i][j] = 0`.
- 변형 **Cherry Pickup** (LeetCode 741) — 2개 경로 동시 진행 시 3D DP `dp[k][i1][i2]` (총 k걸음, 두 사람 행 위치).

### 6.10 프로그래머스 — 등굣길

**요약**: m×n 격자에서 (0,0)→(m-1,n-1) 경로 수. 물웅덩이(puddles)는 못 지나감. mod 10^9+7.
**상태**: `dp[i][j]` = (0,0)→(i,j) 경로 수
**점화식**: 물웅덩이면 0, 아니면 `dp[i][j] = dp[i-1][j] + dp[i][j-1]`

```java
class Solution {
    public int solution(int m, int n, int[][] puddles) {
        int MOD = 1_000_000_007;
        long[][] dp = new long[n + 1][m + 1];
        boolean[][] block = new boolean[n + 1][m + 1];
        for (int[] p : puddles) block[p[1]][p[0]] = true;  // p = [col, row]
        dp[1][1] = 1;
        for (int i = 1; i <= n; i++) {
            for (int j = 1; j <= m; j++) {
                if (i == 1 && j == 1) continue;
                if (block[i][j]) { dp[i][j] = 0; continue; }
                dp[i][j] = (dp[i - 1][j] + dp[i][j - 1]) % MOD;
            }
        }
        return (int) dp[n][m];
    }
}
```


**복잡도**: O(m × n).
**함정**:

- 1-indexed로 짜는 게 base case 깔끔.
- mod 10^9+7 — `long` 안 쓰면 overflow.

### 6.11 프로그래머스 — N으로 표현

**요약**: 숫자 N과 사칙연산만으로 number를 만드는 최소 사용 횟수 (≤ 8).
**상태**: `dp[k]` = N을 k번 사용해 만들 수 있는 수의 집합
**점화식**:
- `dp[k]`에는 NN...N (N을 k번 이어붙인 수)가 포함
- `dp[i]`와 `dp[k-i]`의 모든 조합에 대해 사칙연산 결과를 `dp[k]`에 추가
- k=1부터 8까지 진행, dp[k]에 number가 처음 나타나는 k 반환

```java
class Solution {
    public int solution(int N, int number) {
        List<Set<Integer>> dp = new ArrayList<>();
        dp.add(new HashSet<>());  // dp[0] unused
        for (int k = 1; k <= 8; k++) {
            Set<Integer> set = new HashSet<>();
            // NN...N (N을 k번)
            int repunit = 0;
            for (int i = 0; i < k; i++) repunit = repunit * 10 + N;
            set.add(repunit);
            // dp[i]와 dp[k-i] 조합
            for (int i = 1; i < k; i++) {
                for (int a : dp.get(i)) {
                    for (int b : dp.get(k - i)) {
                        set.add(a + b);
                        set.add(a - b);
                        set.add(a * b);
                        if (b != 0) set.add(a / b);
                    }
                }
            }
            if (set.contains(number)) return k;
            dp.add(set);
        }
        return -1;
    }
}
```

**복잡도**: 상태 수 작음 (k ≤ 8), set 크기에 비례. 실질 O(작음).
**함정**:
- 0으로 나눗셈 체크.
- repunit (N을 k번 이어붙인 수) 빠뜨리면 답 안 나옴.

### 6.12 프로그래머스 — 도둑질 (House Robber II 변형)

**요약**: 원형 배열의 House Robber — 첫 집과 마지막 집이 이웃.
**접근**: 첫 집을 털었으면 마지막 집 못 털고, 안 털었으면 가능 → 두 케이스 비교.
- 케이스 A: `[0, n-2]`에서 일반 House Robber (마지막 집 제외)
- 케이스 B: `[1, n-1]`에서 일반 House Robber (첫 집 제외)
- 답 = `max(A, B)`

```java
class Solution {
    public int solution(int[] money) {
        int n = money.length;
        if (n == 1) return money[0];
        return Math.max(rob(money, 0, n - 2), rob(money, 1, n - 1));
    }

    private int rob(int[] money, int lo, int hi) {
        int prev2 = 0, prev1 = 0;
        for (int i = lo; i <= hi; i++) {
            int curr = Math.max(prev1, prev2 + money[i]);
            prev2 = prev1;
            prev1 = curr;
        }
        return prev1;
    }
}
```

```kotlin
private fun rob(money: IntArray, lo: Int, hi: Int): Int {

        prev1 = curr
    }
    return prev1
}
```

**복잡도**: O(n) 시간, O(1) 공간.
**함정**:
- n=1 단일 원소.
- "원형"의 처리를 두 번 풀기로 분리하는 게 핵심 트릭.

## 7. 함정·엣지케이스

### 7.1 점화식 base case 누락

**증상**: `dp[0]`, `dp[1]` 초기화를 까먹어 무한 루프 또는 잘못된 값.
**예시**: Climbing Stairs에서 `dp[0] = 0`으로 하면 `dp[2] = dp[1] + dp[0] = 1 + 0 = 1`로 오답.
**해결**: 점화식을 적용할 수 없는 최소 부분 문제는 **수기로 직접 계산**. n=0, n=1, m=0, j=0 같은 경계는 항상 의심.

### 7.2 인덱스 오류 (0-indexed vs 1-indexed)

**증상**: `dp[i-1][j-1]` 접근 시 `i=0`이면 `dp[-1][-1]` ArrayIndexOutOfBounds.
**해결**: 2D DP는 거의 대부분 `dp[m+1][n+1]`로 1-indexed로 설계. `dp[0][*]`, `dp[*][0]` 을 base case로 두면 안전.

### 7.3 Overflow

**증상**: Fibonacci, Knapsack의 dp 값이 `int` 범위를 넘김.
**해결**:
- 답이 `2^31 - 1` 근처면 `long` 사용.
- mod 연산 요구되면 **매 단계마다** mod 적용. 마지막에만 mod하면 이미 overflow된 후.
- Edit Distance에서 `INF = Integer.MAX_VALUE`로 두고 `INF + 1`을 더하면 overflow → `Integer.MAX_VALUE / 2` 또는 `n + m + 1` 사용.

### 7.4 메모이제이션의 dirty 값

**증상**: Top-down에서 `memo[n] = 0`으로 초기화했는데, 0이 유효한 답이라 "계산했는지 미계산인지" 구분 못 함.
**해결**:
- `memo`를 `-1`로 초기화 (음수가 답이 될 수 없는 경우).
- 또는 `Integer[]`로 두고 `null` 체크.
- 또는 `boolean[] computed` 별도 배열.

### 7.5 상태 정의 오류

**증상**: 점화식이 자꾸 꼬이거나, 차원이 폭발하거나, 답을 어디서 읽어야 할지 모름.
**해결**:
- 상태 정의를 **문장으로 정확히** 적어본다. "`dp[i]` = i까지 봤을 때 LIS 길이" vs "`dp[i]` = i를 마지막으로 포함하는 LIS 길이"는 다른 점화식.
- LIS 같은 경우 후자가 점화식이 자연스럽고, 답은 `max(dp[*])`로 읽음.

### 7.6 Knapsack 0-1 vs Unbounded 혼동

**증상**: 0-1인데 정방향 순회해서 같은 물건을 여러 번 담음 → 답이 과대.
**해결**:
- **0-1**: 1D에서 무게 **역방향** 순회 (W → weight[i]).
- **Unbounded**: 1D에서 무게 **정방향** 순회.
- 외부/내부 loop 순서도 영향 — Coin Change II (조합 vs 순열) 참고.

### 7.7 Coin Change에서 조합 vs 순열

**증상**: "동전으로 amount를 만드는 방법의 수"에서 답이 너무 큼 → 순서를 다른 것으로 카운팅한 경우.
**해결**:
```java
// 조합 (순서 무관) — coin이 outer
for (int coin : coins)
    for (int a = coin; a <= amount; a++)
        dp[a] += dp[a - coin];

// 순열 (순서 유관) — amount가 outer
for (int a = 1; a <= amount; a++)
    for (int coin : coins)
        if (coin <= a) dp[a] += dp[a - coin];
```

### 7.8 LIS 엄격 증가 vs 비감소

**증상**: 비슷한 LIS인데 답이 다름.
**해결**:
- 엄격 증가 (`<`): O(n log n)에서 **lowerBound** (첫 ≥ 위치를 찾아 덮어쓰기).
- 비감소 (`≤`): O(n log n)에서 **upperBound** (첫 > 위치를 찾아 덮어쓰기).
- 엄격 감소는 배열 뒤집어 LIS.

### 7.9 큰 W의 Pseudo-polynomial

**증상**: Knapsack에서 W가 10^9 같이 크면 TLE/MLE.
**해결**:
- DP는 O(n × W)라 W가 입력 비트 수가 아닌 값에 비례 → W 크면 못 풀림.
- 대안: 가치 v가 작으면 `dp[v]` (가치 기준 DP), 또는 그리디·branch and bound.

### 7.10 음수/0 처리

**증상**: Coin Change에 음수 동전, House Robber에 0 또는 음수 값.
**해결**:
- 문제 제약을 정확히 읽기. 보통 `nums[i] >= 0`.
- 음수 가능한 경우 (예: Maximum Subarray) — `best = nums[0]`로 시작.

## 8. 꼬리질문 트리

### 8.1 "Top-down과 Bottom-up 중 뭘 선호하나? 왜?"

**모범 답**:
- 두 방식은 본질적으로 같다 — 상태 공간을 한 번씩 계산.
- **Top-down 선호 케이스**:
  - 상태 공간은 큰데 실제 도달하는 상태가 적을 때 (게임 트리, 일부 그래프 DP).
  - 점화식이 자연스러운 재귀로 표현될 때 (트리 DP).
  - 코드를 점화식과 1:1로 빠르게 작성할 때.
- **Bottom-up 선호 케이스**:
  - 모든 상태를 어차피 다 계산해야 할 때 (Knapsack, LIS).
  - 공간 최적화 (이전 행만 유지)가 중요할 때.
  - 재귀 깊이가 stack overflow 위험할 때 (n=10^5).
  - Cache friendly가 중요할 때 (밀집 배열 접근).
- **JVM 관점**: Top-down은 함수 호출 오버헤드 + Integer 박싱(HashMap memo) 비용. Bottom-up은 primitive 배열로 cache hit 좋음.

### 8.2 "공간 복잡도를 O(n)에서 O(1)로 줄이려면?"

**모범 답**:
- `dp[i]`가 `dp[i-1]`, `dp[i-2]` 같은 **상수 개 이전 값**만 참조하면 변수 2~3개로 압축 가능.
- 2D DP `dp[i][j]`가 `dp[i-1][...]`만 참조하면 행 2개로 압축, 더 나아가 1D + 역방향 순회로 1행만 사용.
- 트레이드오프: 코드 가독성 손해, 디버깅 어려움. 면접관이 "더 줄일 수 있나?" 물으면 본질 이해를 보여주는 기회.

### 8.3 "LIS O(n log n) 알고리즘의 원리를 설명하라"

**모범 답**:
- **Patience Sort 비유**: 카드를 한 장씩 보며 더미(pile)에 놓되, 자기보다 작은 더미 위에만 놓는다. 더미가 없으면 새로 만든다. 더미 개수 = LIS 길이.
- `tails[k]` = 길이가 k+1인 모든 증가 부분 수열 중 **마지막 원소가 가장 작은 것의 마지막 값**.
- 새 원소 x를 보면 `tails`에서 x 이상인 첫 위치를 binary search(lowerBound) → 그 위치 덮어쓰기. lowerBound 결과가 size면 size 증가.
- `tails`는 **항상 정렬된 상태 유지** (귀납법 증명 가능) → binary search 가능 → O(log n).
- 주의: `tails` 자체는 실제 LIS가 아니다 (길이만 같음).

### 8.4 "Knapsack에서 0-1과 무한 개수의 차이는?"

**모범 답**:
- **0-1**: 각 물건을 0번 또는 1번만 사용. 1D DP에서 무게를 **역방향** 순회 → 같은 행에서 같은 물건 재사용 방지.
- **Unbounded**: 각 물건을 0번 이상 무한 사용 가능. 1D DP에서 무게를 **정방향** 순회 → 같은 행에서 같은 물건 여러 번 사용.
- 점화식은 같지만 순회 방향이 다르다. 2D로 풀면 차이 없음 (`dp[i-1]` 명시적으로 참조하므로).
- **Bounded Knapsack** (제한 개수): "물건 i를 최대 c_i개" — 이항 분해(binary decomposition)로 O(n × W × log c) 또는 monotonic queue로 O(n × W).

### 8.5 "Bitmask DP는 언제 쓰나? 한계는?"

**모범 답**:
- **언제**: 상태 중 하나가 "방문한 집합" 또는 "선택한 부분집합"일 때, 그리고 **n ≤ 20** 정도.
- 상태 수가 `2^n × n` 이상이므로 n=20이면 약 `2^20 × 20 = 2천만`, 시간 O(2^n × n²)면 4억 — 빠듯하지만 가능.
- **대표 문제**: TSP, "모든 부분집합을 봐야 하는" 문제, 비트당 하나의 작업/도시/사람 매칭 (Assignment problem).
- **한계**: n > 20이면 2^n이 폭발 → 다른 접근 (Held-Karp는 동일, 그래서 NP-hard).
- **운영 응용**: PostgreSQL/Oracle의 query optimizer가 5~10개 테이블 join order 결정 시 Bitmask DP 사용. n이 커지면 휴리스틱(genetic, greedy)으로 fallback.

### 8.6 "DP와 Greedy의 차이는?"

**모범 답**:
- **Greedy**: 매 단계의 지역 최적이 전역 최적임을 **증명할 수 있을 때만** 사용. 정렬 후 그리디가 정석. 예: Activity Selection, Huffman, Dijkstra (음수 간선 없을 때).
- **DP**: 지역 최적이 전역 최적을 보장 안 함 → 모든 선택지를 탐색 후 best 선택. 예: Coin Change.
- **반례 — Coin Change**: `coins = [1, 3, 4], amount = 6` → 그리디(큰 동전부터)는 4+1+1=3개, DP는 3+3=2개. 그리디로 풀리지 않는 케이스.
- **운영**: Greedy가 더 빠르지만 증명 필요. 증명 못 하면 DP가 안전. 면접에서 "왜 그리디로 풀려요?" → "exchange argument" 또는 "matroid 구조"로 답변.

### 8.7 "DP의 시간 복잡도가 너무 클 때 어떻게 줄이나?"

**모범 답**:
1. **상태 압축**: 불필요한 차원 제거. 예: `dp[i][j]`에서 j가 0 또는 1뿐이면 차원 축소.
2. **전이 비용 줄이기**: 점화식 안의 max/min/합을 자료구조(monotonic deque, segment tree, BIT)로 가속.
   - 예: LIS DP의 `max(dp[j])` for j < i with `nums[j] < nums[i]` → BIT로 O(log n) → 전체 O(n log n).
3. **상태 자체 줄이기**: 문제 변환. 예: subset sum을 meet in the middle로 O(2^(n/2)).
4. **다른 패턴으로 전환**: DP가 너무 느리면 그리디 증명 시도, 또는 수학적 닫힌 형태 발견 (Fibonacci 행렬 거듭제곱 O(log n)).

### 8.8 "DP 문제 디버깅 어떻게?"

**모범 답**:
- **테이블 출력**: 작은 입력으로 dp 테이블 전체를 찍어 손계산과 비교.
- **base case 분리 확인**: `dp[0]`, `dp[1]` 등을 hardcode.
- **점화식 단위 테스트**: 특정 `dp[i][j]` 값이 예상과 같은지 assert.
- **Top-down으로 변환**: 재귀로 풀고 호출 흐름 출력 → 어디서 잘못된 값이 들어가는지 추적.
- **상태 정의 재검증**: "정확히 무엇인가?" 문장으로 적고 점화식과 일치 확인.

## 9. 다른 패턴과의 연결

### 9.1 Backtracking + Memo = DP

```
Backtracking (10장):
   solve(state):
       if base: return
       for choice in choices:
           apply(choice)
           solve(newState)
           undo(choice)

   → 모든 경로 탐색, 같은 state 재방문 시 재계산

DP (Top-down):
   solve(state):
       if base: return baseValue
       if memo[state] exists: return memo[state]
       result = best of solve(newState) for all choices
       memo[state] = result
       return result

   → 같은 state는 한 번만 계산, 결과 재사용
```

**핵심**: 백트래킹에서 같은 state 재계산이 일어나면 DP로 전환. 차이는 memo + return 값.

### 9.2 DFS + Memo = Top-down DP

트리/그래프 DP는 거의 항상 DFS + memoization.
- 트리 DP: 각 노드에서 자식 결과 합산. memo 불필요(트리는 부분 문제 안 겹침).
- DAG에서의 DP: 위상정렬 후 순회 = bottom-up DP. 또는 DFS + memo = top-down DP.
- 일반 그래프 DP (Bellman-Ford): "k번 이하 간선으로 도달" 상태 — 사이클 때문에 단순 DFS 불가.

### 9.3 Greedy vs DP의 경계

| 문제 | Greedy 가능? | 이유 |
|---|---|---|
| Activity Selection | O | "종료 시간 빠른 것" 정렬 → exchange argument |
| Coin Change (일반) | X | 반례 존재 (`[1,3,4], 6`) → DP |
| Coin Change (canonical: 한국 동전) | O | 화폐 시스템이 canonical → 큰 동전부터 |
| Fractional Knapsack | O | 가치/무게 비 정렬 → exchange |
| 0-1 Knapsack | X | 쪼갤 수 없음 → DP |
| Huffman | O | "최소 빈도 2개 합치기" greedy 증명됨 |
| LIS | X | 단순 그리디로 안 됨 → DP |

**판단법**: 그리디 선택의 안전성(safe choice)을 exchange argument로 증명 시도 → 안 되면 DP.

### 9.4 Divide & Conquer vs DP

| | D&C | DP |
|---|---|---|
| 부분 문제 | 안 겹침 | 겹침 |
| 예시 | Merge sort, Quick sort, FFT | LCS, Knapsack |
| Memo | 불필요 | 필수 |
| 시간 | T(n) = 2T(n/2) + O(n) = O(n log n) | 상태 수 × 전이 |

**연결**: D&C로 시작해서 부분 문제가 겹친다는 걸 발견하면 → DP로 전환. 예: Fibonacci를 단순 재귀로 풀면 O(2^n), memo 추가하면 O(n).

### 9.5 Binary Search + DP (Parametric / Aliens Trick)

- "이 값보다 작은 답이 가능한가?"를 결정 문제로 변환 → 답을 binary search.
- DP를 결정 문제 안에서 호출.
- 예: "k개의 부분 배열로 나눌 때 최대합의 최솟값" (LeetCode 410) — 답을 binary search, 각 mid에 대해 DP/그리디로 검증.

### 9.6 운영 시스템에서의 DP 패턴

| 문제 | DP 모델 | 운영 의미 |
|---|---|---|
| **Query optimizer join order** | Bitmask DP, `dp[S]` = S 집합의 최소 join 비용 | PostgreSQL/Oracle, N개 테이블 join 시 최적 순서. N ≥ 12면 휴리스틱 fallback |
| **Cache replacement (Belady)** | "미래에 가장 늦게 쓰일 항목 제거" — DP로 OPT 계산 (lower bound) | LRU/LFU 평가 벤치마크 |
| **Bin Packing (K8s scheduler)** | First-fit decreasing은 그리디, exact는 0-1 Knapsack 다차원 | Pod 자원 할당 |
| **Sequence Alignment** | Needleman-Wunsch = Edit Distance 일반화 (가중치 행렬) | DNA, NLP spell checker |
| **Routing (Bellman-Ford)** | `dp[k][v]` = k번 이하 간선으로 v 도달 최단 거리 | OSPF 변형, BGP path |
| **Compiler instruction scheduling** | DAG-based DP (register pressure 최소화) | gcc/llvm 백엔드 |
| **Investment portfolio** | 다차원 Knapsack (가치, 위험, 유동성) | 금융 최적화 |

→ DP는 "코딩 테스트용 알고리즘"이 아니라 **자원 최적화·매칭·비교가 필요한 모든 시스템의 공통 도구**. 마스터는 이 매핑을 자유롭게 한다.

## 11. 더 깊이

- **CLRS Chapter 14, 15** — Dynamic Programming의 표준 교과서. Matrix Chain, LCS, Optimal BST.
- **Erickson, *Algorithms*** — 무료 PDF, DP 챕터의 점화식 도출 과정이 명료.
- **Selinger et al. (1979)** — "Access Path Selection in a Relational Database Management System". PostgreSQL/Oracle query optimizer의 Bitmask DP 원조.
- **Belady (1966)** — "A study of replacement algorithms for a virtual-storage computer". OPT 알고리즘.
- **Needleman & Wunsch (1970)** — DNA sequence alignment, Edit Distance의 생물학 원형.
- **Held & Karp (1962)** — TSP의 Bitmask DP 원조, O(2^n × n²).
- **Knuth — *The Art of Computer Programming Vol. 3*** — 패턴 매칭과 정렬에서 DP 관점.
- **Competitive Programmer's Handbook (Antti Laaksonen)** — Chapter 7 (Dynamic Programming), 점화식 catalog.
