# resume_v13 읽기성 피드백

대상: [resume_v13.md](./resume_v13.md)

이 문서는 이력서의 기술적 진위를 평가하는 문서가 아니라, 외부 채용 담당자나 개발자가 실제로 읽을 때 **첫 스캔에서 이해되는지**를 기준으로 적었습니다. 결론부터 말하면, 지금 원고는 내용 자체보다 **문장 밀도와 표현 방식** 때문에 읽기 부담이 큽니다. 특히 `문제` 설명이 지나치게 길고, 한 문단 안에 사례·제약·원인·해결·성과가 동시에 들어가며, 같은 의미를 다른 말로 여러 번 되풀이하는 구간이 많습니다.

## 전체 평가

현재 원고의 강점은 분명합니다. 운영 문제를 단순 기능 설명이 아니라 `왜 어려웠는지`까지 풀어 쓰고 있고, 수치와 결과도 많이 붙어 있습니다. 다만 이력서에서는 그 강점이 그대로 장점으로만 작동하지 않습니다. 외부 독자는 한 사례를 10~20초 안에 스캔하는 경우가 많아서, 정보가 많아도 **구조가 먼저 보이지 않으면** 내용이 전달되지 않습니다.

가장 큰 문제는 세 가지입니다.

1. 한 문장에 정보가 너무 많다.
2. 한 문단 안에서 같은 개념을 반복 설명한다.
3. 핵심 사례보다 용어가 먼저 보인다.

그래서 읽는 사람 입장에서는 “무슨 문제를 풀었는가”보다 “어떤 전문 용어를 많이 아는가”가 먼저 남을 수 있습니다.

## 공통적으로 보이는 읽기 어려운 패턴

### 1. 한 문장 안에 사례가 3개 이상 들어감

대표 예시는 사례 2의 `좀비 schedule`, `중복 빌링키`, `admin override orphan`입니다. 각각 자체로는 좋은 사례인데, 지금은 한 문장 안에 다 들어가 있어서 첫 독해가 무겁습니다.

문제점:

- 독자가 첫 줄에서 구조를 못 잡는다.
- 각 사례의 차이가 흐려진다.
- 뒤의 `eventual consistency` 설명까지 한 번에 처리해야 해서 피로도가 높다.

고치는 방법:

- 사례는 무조건 줄바꿈으로 분리한다.
- 첫 줄은 “무슨 종류의 문제인지”만 말한다.
- 두 번째 줄부터 각 사례를 bullet로 나눈다.
- `조용한 결함` 같은 표현은 한 번만 쓰고, 반복하지 않는다.

권장 형태:

```md
**문제** — 외부 PG 상태와 DB 상태가 어긋나면서 3종의 정합성 오류가 발생했습니다.
- 좀비 schedule: ...
- 중복 빌링키: ...
- admin override orphan: ...

이 문제들은 사용자 클레임이나 환불 분쟁이 발생한 뒤에야 드러나는 조용한 결함이었습니다.
```

### 2. 문제 정의와 원인 설명이 한 문단에 섞임

사례 2, 3, 10, 11, 15에서 자주 보입니다. `무슨 일이 있었는지`와 `왜 어려운지`와 `어떤 제약이 있었는지`가 한 줄 안에서 섞입니다.

문제점:

- 독자는 원인과 결과를 분리해 읽을 수 없다.
- “그래서 이 사람이 실제로 한 일이 뭐지?”가 늦게 보인다.
- 채용자가 훑어볼 때 스킵될 가능성이 높다.

고치는 방법:

- `문제`는 현상과 피해만 쓴다.
- `해결`에서 제약과 설계를 쓴다.
- `성과`에서 숫자와 결과를 쓴다.
- `문제` 문단 안에는 구현 디테일을 넣지 않는다.

### 3. 같은 말을 동의어로 여러 번 반복함

예를 들면 `eventual inconsistency`, `DB 와 PG 의 의미 차이`, `billing 단위 / schedule 단위`, `조용한 결함`이 같은 문단에서 반복됩니다. 의미는 맞지만, 이력서에서는 중복으로 읽힙니다.

문제점:

- 읽는 속도가 느려진다.
- 문장이 장황해 보인다.
- “설명은 많은데 핵심은 압축되지 않았다”는 인상을 준다.

고치는 방법:

- 핵심 표현은 하나만 남긴다.
- 같은 의미를 두 번 설명하지 않는다.
- 기술 용어는 `한 번 정의하고 이후에는 약어처럼 사용`한다.

예:

```md
DB와 PG의 상태가 다른 문제였습니다.
```

이 한 줄 뒤에 굳이 `billing 단위`, `schedule 단위`, `eventual consistency`를 모두 같은 문단에서 반복할 필요는 없습니다. 뒤쪽 `해결`에서 필요할 때만 다시 꺼내면 됩니다.

### 4. 수식어가 너무 많아 핵심 동사가 늦게 나온다

예: `결제 운영의 정석인`, `의심 대상 K << N 으로 압축`, `4-way 분기 자동 정정`, `정확한 조건의 단일 SQL`, `시니어 데코레이터 패턴` 같은 표현이 한 줄에 겹칩니다.

문제점:

- 핵심 동작보다 포장 문구가 앞선다.
- 문장이 자기평가처럼 읽힐 수 있다.
- 외부 독자에게는 과장으로 보일 위험이 있다.

고치는 방법:

- 수식어는 1개만 남긴다.
- 나머지는 성과나 설명 쪽으로 옮긴다.
- “정석”, “시니어”, “본질적” 같은 평가어는 최소화한다.

권장 방식:

```md
의심 대상만 추려 외부 PG API 호출을 최소화했고, 결과를 4-way로 분기해 자동 정정했습니다.
```

이 한 줄이면 기능이 바로 보입니다.

### 5. 괄호와 부연 설명이 너무 많음

원고 전체에서 괄호 안 설명이 굉장히 많습니다. 경우에 따라서는 유용하지만, 한 문장 안에 2~3개가 쌓이면 숨이 막힙니다.

문제점:

- 본문과 부연 설명의 경계가 흐려진다.
- 읽는 리듬이 끊긴다.
- 중간에 멈춰 다시 해석해야 한다.

고치는 방법:

- 괄호는 정의용으로만 제한한다.
- 구현 디테일은 별도 bullet 또는 다음 문장으로 뺀다.
- 괄호 안에 또 괄호를 넣지 않는다.

## 섹션별 피드백

### 사례 1~3: 결제 도메인

이 구간은 내용이 좋지만, 첫 세 사례가 모두 `문제 → 해결 → 성과` 구조를 가지면서도 각 문단이 지나치게 길어 비슷하게 느껴집니다.

개선 포인트:

- 사례 2는 특히 긴 편이라 3개 하위 bullet로 분리하는 것이 좋습니다.
- 사례 3은 `사전 검증 / 트랜잭션 / schedule fallback` 3단계가 강점이므로, 해결에서 번호를 그대로 살리되 문장은 짧게 끊는 편이 좋습니다.
- 사례 1의 `4중 방어선` 설명은 강하지만, 한 문장 안에 모든 방어선을 다 넣으면 처음 읽는 사람이 놓칩니다. 방어선은 bullet이 더 적합합니다.

추천 수정 방향:

```md
**문제** — ...
**해결** — ...
- 사전 검증
- 트랜잭션
- schedule fallback 보정
**성과** — ...
```

### 사례 4~8: 대량 처리 / 운영 / 대시보드

이 구간은 상대적으로 구조가 안정적입니다. 다만 여전히 각 사례에서 `문제` 문장이 너무 길고, `성과`에 숫자와 설명이 한 줄씩 붙어 있어 빠르게 훑기 어렵습니다.

개선 포인트:

- `문제`는 2문장 이내로 줄인다.
- `성과`는 숫자 2~3개만 남기고 나머지는 한 줄로 묶는다.
- `JVM/Spring 매핑`은 너무 길면 핵심 기술 3개만 남긴다.

예를 들어 사례 6의 `13,127 매장 / 5,719 platform_id 그룹 / 13,699 task`는 인상적이지만, 이력서에서는 모두 동일 중요도로 보이지 않을 수 있습니다. 가장 임팩트 있는 숫자 1~2개만 전면에 두는 편이 좋습니다.

### 사례 9~10: 세션 락 / IP 평판

이 구간은 운영 난이도와 기술 설계가 잘 드러납니다. 다만 `문제`와 `해결` 모두 길고 용어가 많아서, 읽는 사람은 여기서 피로를 느끼기 시작할 가능성이 큽니다.

개선 포인트:

- `cold-start deadlock`, `CROSSSLOT`, `port rotation 회피`처럼 중요한 위험은 남기되, 각 용어마다 설명을 한 번만 붙인다.
- `핵심 결정 4가지`처럼 구조화할 수 있는 부분은 더 적극적으로 bullet화한다.
- `성과`는 숫자와 사고 방지 결과를 분리해서 보여주는 것이 좋다.

권장:

```md
**문제** — ...
**해결** — 핵심 결정 4가지
1. ...
2. ...
3. ...
4. ...
```

### 사례 11~12: 우회 / 포트화

이 구간은 기술적 깊이가 드러나지만, 독자 친화성은 가장 약한 편입니다. 특히 `discriminated union`, `spec 헤더`, `세대 가드`, `sync throw 정규화` 같은 개념이 한 덩어리로 나오면, 배경지식이 없는 외부 독자는 따라가기 어렵습니다.

개선 포인트:

- 행동 계약은 표로 두고, 본문은 짧게 유지한다.
- `왜 이 패턴이 필요한가`와 `무엇을 보장하는가`를 분리한다.
- 너무 세밀한 구현 단위는 이력서 본문이 아니라 부록이나 기술 블로그로 넘기는 것이 낫다.

특히 사례 12는 지금도 구조가 나쁘지 않지만, `문제`에서 미묘한 invariant 를 한 줄에 너무 많이 밀어넣습니다. 여기서는 `첫 구현의 문제`와 `새 구조의 보장`만 남기고, 세부 계약은 표로만 두면 훨씬 읽기 쉬워집니다.

### 사례 13~15: AI / 자동화

이 구간은 숫자와 알고리즘 설명이 많아 좋기도 하지만, 동시에 가장 설명 과잉으로 읽힐 위험이 있습니다. 특히 `6신호`, `tanh`, `Bell`, `TOCTOU 4단 게이트`, `API Key → JWT fallback` 같은 요소가 한 사례 안에 많이 들어갑니다.

개선 포인트:

- 알고리즘 이름은 남기되, 각 이름의 정의를 본문에 너무 길게 쓰지 않는다.
- `문제`는 운영상 왜 필요한지만 쓴다.
- `해결`은 핵심 알고리즘 이름 2~3개만 남기고, 나머지는 기술 스택이나 부록으로 이동한다.

특히 사례 13은 `신호 처리 + 정보 이론 관점`이라는 설명이 멋있지만, 채용자가 원하는 건 수식의 미학보다 “이 시스템이 실제로 어떤 운영 문제를 줄였는가”입니다. 따라서 해설보다 효과를 더 전면에 두는 편이 낫습니다.

## 문장 단위로 특히 손볼 부분

아래 같은 문장들은 내용이 나쁜 게 아니라, 이력서 문장으로 너무 길고 무거운 편입니다.

- 사례 2의 첫 `문제` 문장
- 사례 10의 `문제` 문장
- 사례 11의 `문제` / `해결` 문장
- 사례 12의 `문제` 문장
- 사례 15의 `문제` 문장

이들 문장은 거의 공통적으로 아래 문제를 가집니다.

1. 현상 설명이 끝나기 전에 원인 분석이 시작된다.
2. 원인 설명이 끝나기 전에 해결 방법이 시작된다.
3. 해결 방법이 끝나기 전에 성과가 암시된다.

이 구조는 글쓴이 입장에서는 자연스럽지만, 읽는 사람 입장에서는 부담스럽습니다.

## 추천하는 전반적 수정 원칙

### 1. 한 사례당 첫 문장은 1개 문제만 말한다

좋은 예:

```md
외부 PG 상태와 DB 상태가 어긋나 정기결제 정합성 오류가 발생했습니다.
```

나쁜 예:

```md
외부 PG 상태와 DB 상태가 어긋나고, 호출량 제약이 있고, billing 단위와 schedule 단위가 달라서, 사용자 클레임 뒤에야 드러나는 eventual inconsistency가 발생했습니다.
```

### 2. 설명은 `문제 / 해결 / 성과`로만 이동시킨다

각 문단의 역할을 고정하면 문장 길이를 줄이기 쉬워집니다.

- 문제: 현상
- 해결: 설계
- 성과: 결과

### 3. 세부 용어는 처음 한 번만 정의한다

`SCHEDULED`, `schedule_status`, `billing 단위`, `schedule 단위` 같은 용어는 한 번만 정의하고, 이후에는 반복하지 않는 편이 낫습니다.

### 4. bullet을 더 적극적으로 쓴다

지금 원고는 본문이 너무 서술형입니다. 이력서에서는 bullet이 단순한 목록이 아니라, 독자가 정보를 소화하는 단위입니다.

### 5. 평가어를 줄이고 사실어를 늘린다

`정석`, `본질적`, `시니어`, `정확한`, `조용한 결함` 같은 표현은 적절할 때만 사용하고, 나머지는 사실만 쓰는 편이 안정적입니다.

## 우선 수정 순서

1. 사례 2, 10, 11, 12, 15의 `문제` 문장을 먼저 쪼갠다.
2. 각 사례의 `해결`에서 괄호 부연을 줄이고 bullet로 이동한다.
3. `성과`는 숫자 중심으로 재배열한다.
4. `JVM/Spring 매핑`은 핵심 기술만 남기고 압축한다.
5. 마지막으로 문서 전체에서 중복 표현을 제거한다.

## 한 줄 결론

지금 원고는 기술력은 보이지만, 외부 독자가 읽기에는 과밀합니다. 문장 자체를 더 잘 쓰는 문제라기보다, **정보를 더 잘 쪼개는 문제**로 보는 게 맞습니다. 지금 상태에서는 좋은 내용이 많아도 읽는 사람이 따라오기 어렵고, 채용 문서로는 그 손해가 큽니다.
